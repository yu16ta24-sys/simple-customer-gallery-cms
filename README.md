## 【Geminiへのレビュー依頼事項】

今回の v1.1.0 は、ブログ/お知らせ管理の最小実装フェーズです。
ギャラリー機能は壊さず、横並びで `SCG_Content_Post_Type` と `SCG_Content_Manage` を追加しています。

重点的に見てほしい点：

1. CPT設計が適切か
   - `scg_blog` / `scg_news` を分離した設計
   - `show_ui => false` にしてWordPress標準投稿画面を見せない設計
   - 将来フロント表示やショートコード追加をする前提で破綻しないか

2. ブログ/お知らせ管理の保存処理が安全か
   - `admin_post_scg_save_content`
   - nonceチェック
   - 権限チェック
   - `wp_insert_post` / `wp_update_post`
   - 画像アップロード処理

3. 添付画像3枚の扱いが妥当か
   - `_scg_image_1`〜`_scg_image_3` のmeta設計
   - 画像削除時に attachment 自体は残し、metaだけ外す設計
   - 完全削除時のみ attachment も削除する設計

4. 削除仕様が適切か
   - 通常削除は `_scg_status = hidden`
   - 削除済み一覧から復元可能
   - 管理者のみ完全削除

5. UI/UXが「迷わず操作できる専用CMS」として自然か
   - ブログ/お知らせで同じUIを使うことに違和感がないか
   - 公開/下書きの選択UI
   - 添付画像を最大3枚まで追加するUI
   - 一覧画面の見やすさ

6. 次フェーズでフロント表示を実装する前に、追加で整理すべき点があるか

特に問題なければ「このまま次フェーズへ進んでOK」と判断してください。
改善点があれば、優先度付きで具体的に指摘してください。

## データ構造仕様

### scg_photo

ギャラリー画像は、WordPressのカスタム投稿タイプ `scg_photo` として1画像につき1投稿で管理する。

meta:

- `_scg_attachment_id` 実画像のattachment_id
- `_scg_order` 同一サブカテゴリ内での表示順。型はint想定
- `_scg_status` `active` / `hidden`
- `_scg_uploaded_by` アップロードしたユーザーID

taxonomy:

- `scg_gallery_category`
  - 親カテゴリ = メインカテゴリー
  - 子カテゴリ = サブカテゴリー

並び順:

- 同一サブカテゴリ内で `_scg_order` 昇順

---

### scg_blog / scg_news

ブログとお知らせは、それぞれ独立したCPTで管理する。

```txt
scg_blog = ブログ
scg_news = お知らせ
```

基本データ:

```txt
post_title   = タイトル
post_content = 本文
post_status  = publish / draft
post_author  = 投稿者
```

meta:

```txt
_scg_status = active / hidden
_scg_image_1 = attachment_id
_scg_image_2 = attachment_id
_scg_image_3 = attachment_id
```

削除:

- 通常削除は `_scg_status = hidden`
- 復元時は `_scg_status = active`
- 完全削除は管理者のみ
- 完全削除時は添付画像も削除する

## 設計上の判断基準

このプラグインは、コード上はWordPressの投稿・メディア・タクソノミーを利用しているが、設計上は「お客様用の専用CMS」である。

そのため、判断基準は以下。

```txt
WordPressとして正しいか
よりも
お客様が迷わず操作できるか
```

優先順位:

```txt
1. 誤操作しにくい
2. 画面が分かりやすい
3. 動作が気持ちいい
4. WordPress標準機能を壊さない
5. 将来拡張しやすい
```
